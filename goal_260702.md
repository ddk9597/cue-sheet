아래 “회원 전용 기능 MVP 구현 계획”을 기준으로 실제 구현을 진행하라.

중요:

* 계획을 다시 크게 재작성하지 말고, 현재 프로젝트 구조를 확인한 뒤 이 계획에 맞춰 구현하라.
* 기존 기능을 깨지 않는 최소 변경으로 구현하라.
* 새 라이브러리 추가는 피하고, 기존 코드 스타일/DB 유틸/인증 방식/라우팅 방식을 우선 사용하라.
* 구현 중 계획과 실제 코드 구조가 충돌하면 임의로 우회하지 말고, 가장 안전한 최소 변경안을 적용하고 마지막 요약에 이유를 남겨라.

# 회원 전용 기능 MVP 구현 계획

## Summary

* 현재 프로젝트는 정적 HTML/CSS/JS + Vercel Serverless API + Neon Postgres 구조이며, 인증은 `cue_sheet_session` HttpOnly 쿠키와 `getSessionUser(sql, request)`로 처리한다.
* 현재 서버리스 함수는 8개이므로 Hobby 제한 12개 이하다. 그래도 새 API 파일은 최소화하고, 회원 전용 API는 `api/member/[...path].js` catch-all 1개로만 추가한다.
* “메시지”는 일반 채팅/쪽지가 아니라 `group_invites`를 메시지함 형태로 보여주는 초대 알림으로 구현한다.

## DB 설계

`api/_lib/db.js`의 `ensureSchemaLocked()`에 idempotent DDL을 추가한다.

추가 테이블:

1. `groups`

   * `id BIGSERIAL PRIMARY KEY`
   * `name TEXT NOT NULL DEFAULT ''`
   * `owner_user_id BIGINT NOT NULL REFERENCES app_users(id) ON DELETE CASCADE`
   * `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`
   * `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`
   * index: `groups_owner_user_id_idx`

2. `group_members`

   * `group_id BIGINT NOT NULL REFERENCES groups(id) ON DELETE CASCADE`
   * `user_id BIGINT NOT NULL REFERENCES app_users(id) ON DELETE CASCADE`
   * `role TEXT NOT NULL DEFAULT 'member' CHECK (role IN ('owner', 'member'))`
   * `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`
   * primary key: `(group_id, user_id)`
   * index: `group_members_user_id_idx`

3. `group_invites`

   * `id BIGSERIAL PRIMARY KEY`
   * `group_id BIGINT NOT NULL REFERENCES groups(id) ON DELETE CASCADE`
   * `inviter_user_id BIGINT NOT NULL REFERENCES app_users(id) ON DELETE CASCADE`
   * `invitee_user_id BIGINT NOT NULL REFERENCES app_users(id) ON DELETE CASCADE`
   * `status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'accepted', 'rejected'))`
   * `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`
   * `responded_at TIMESTAMPTZ`
   * partial unique index: `(group_id, invitee_user_id) WHERE status = 'pending'`

4. `user_memos`

   * `user_id BIGINT PRIMARY KEY REFERENCES app_users(id) ON DELETE CASCADE`
   * `content TEXT NOT NULL DEFAULT ''`
   * `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`

5. `user_practice_calendar_state`

   * `user_id BIGINT PRIMARY KEY REFERENCES app_users(id) ON DELETE CASCADE`
   * `logs JSONB NOT NULL DEFAULT '{}'::jsonb`
   * `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`

DDL 조건:

* 모든 DDL은 idempotent하게 작성하라.
* `CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS` 등 기존 `ensureSchemaLocked()` 패턴을 따르라.
* 기존 데이터에 영향을 주지 마라.

## Backend Changes

### `api/cues.js`

* 확인 결과 이미 로그인 사용자는 `getSessionUser()` 기준으로 `user_cue_sheet_state`를 사용한다.
* 코드는 유지하되, 수동 테스트로 계정별 분리를 재검증하라.

### `api/practice.js`

* `getSessionUser(sql, request)`를 추가하라.
* 로그인 사용자는 `user_practice_calendar_state`에서 조회/저장하라.
* 비로그인 사용자는 기존 `practice_calendar_state + 저장 비밀번호` 흐름을 유지하라.
* 기존 비로그인 저장 흐름을 깨지 마라.

### `api/member/[...path].js`

회원 전용 API catch-all 파일을 1개만 추가하라.

공통 규칙:

* 모든 route는 먼저 `ensureSchema()` 또는 기존 프로젝트의 schema 보장 함수를 호출하라.
* 이후 `getSessionUser(sql, request)`로 로그인 사용자를 확인하라.
* 로그인 사용자가 없으면 `401 not_authenticated`를 반환하라.
* payload의 `userId`, `memberId`, `accountId`는 신뢰하지 마라.
* 모든 조회/수정/삭제는 반드시 `sessionUser.id` 기준으로 처리하라.
* 트랜잭션은 현재 프로젝트의 DB 유틸 또는 사용 중인 Neon/Postgres 클라이언트가 지원하는 방식에 맞춰 구현하라.
* 임의로 새로운 DB 라이브러리를 추가하지 마라.

구현 route:

1. `GET /api/member/groups`

   * 내가 속한 그룹 목록을 반환하라.
   * 응답에는 현재 사용자의 `role(owner/member)`을 포함하라.
   * 프론트에서 owner 그룹만 초대 가능하게 판단할 수 있어야 한다.

2. `POST /api/member/groups`

   * `{ name }`으로 그룹을 생성하라.
   * `groups` insert와 `group_members(role='owner')` insert를 트랜잭션으로 처리하라.
   * 그룹 생성자는 반드시 owner 멤버로 추가되어야 한다.

3. `POST /api/member/invites`

   * `{ groupId, email }`로 등록된 사용자를 초대하라.
   * 초대 생성은 그룹 owner만 가능하다.
   * 초대 이메일은 `trim()` 및 소문자 정규화 후 조회하라.
   * 존재하지 않는 이메일이면 적절한 오류를 반환하라.
   * 자기 자신 초대는 차단하라.
   * 이미 그룹 멤버인 사용자는 초대하지 마라.
   * 같은 그룹/사용자에 대한 pending 초대가 이미 있으면 중복 생성하지 마라.
   * pending 초대 중복 방지는 Postgres partial unique index도 함께 사용하라.

4. `GET /api/member/messages`

   * `group_invites` 중 `invitee_user_id = sessionUser.id`인 초대 알림 목록을 반환하라.
   * MVP에서는 기본적으로 pending 초대를 메시지함에 표시하라.
   * 응답에는 그룹명, 초대한 사람, 상태, 생성일을 포함하라.
   * 수락 버튼은 pending 상태에만 표시될 수 있도록 필요한 상태값을 반환하라.

5. `POST /api/member/messages/accept`

   * `{ inviteId }`로 초대를 수락하라.
   * 현재 로그인 사용자가 invitee이고, status가 pending인 초대만 수락할 수 있다.
   * `group_members` insert와 `group_invites.status='accepted'`, `responded_at=NOW()` 업데이트를 트랜잭션으로 처리하라.
   * 이미 멤버인 경우 중복 insert가 발생하지 않도록 `ON CONFLICT` 또는 사전 검증을 사용하라.
   * accepted/rejected 상태의 초대는 재수락할 수 없게 하라.
   * invitee가 아닌 사용자는 수락할 수 없게 하라.

6. `GET /api/member/memo`

   * 현재 로그인 사용자의 개인 메모를 조회하라.
   * 이번 MVP에서 메모는 사용자당 하나의 개인 메모장이다.

7. `PUT /api/member/memo`

   * `{ content }`로 현재 로그인 사용자의 개인 메모를 저장하라.
   * `INSERT ... ON CONFLICT (user_id) DO UPDATE` 방식 또는 기존 프로젝트 패턴에 맞는 upsert를 사용하라.

## Frontend Changes

### `index.html`

로그인 영역 아래에 회원 전용 패널을 추가하라.

로그인 전:

* 회원 전용 패널은 숨김 또는 비활성 상태로 표시한다.
* “로그인 후 사용 가능” 상태를 보여줄 수 있다.

로그인 후 패널 구성:

* 내 그룹 목록
* 그룹 만들기 입력
* 초대할 그룹 select
* 초대 이메일 입력
* 초대 보내기 버튼
* 초대 알림 메시지함
* 초대 수락 버튼
* 개인 메모 textarea
* 메모 저장 버튼

### `script.js`

* `MEMBER_API_ENDPOINT = "/api/member"`를 추가하라.
* 로그인 성공 후 `loadMemberWorkspace()`를 실행하라.
* 초기 세션 확인 후 로그인 상태면 `loadMemberWorkspace()`를 실행하라.
* 로그아웃 직후 그룹, 메시지, 메모, 큐시트, 연습 기록 관련 JS 상태와 DOM을 초기화하라.
* 이후 기존 로컬/공용 데이터 로딩 흐름을 실행하라.
* 그룹이 없으면 초대 select, 초대 이메일 입력, 초대 버튼을 비활성화하라.
* owner 권한이 있는 그룹에 대해서만 초대 가능하게 하라.
* 초대 수락 후 그룹 목록과 메시지함을 다시 조회하라.
* 메모는 textarea 하나로 저장/불러오기한다.
* 로그인 사용자의 연습 기록 저장 시 저장 비밀번호 prompt를 띄우지 않고 계정 기준으로 저장하라.
* 비로그인 사용자의 기존 저장 비밀번호 흐름은 유지하라.

### `style.css`

* 기존 `panel`, `field`, `primary-button`, `ghost-button`, `save-status` 스타일을 재사용하라.
* 별도 랜딩/대형 UI 없이 메인 페이지의 관리 패널 톤으로 구현하라.
* 카드 중첩을 피하고 기존 UI와 이질감이 없게 구성하라.

## Test Plan

구현 후 아래 검증을 수행하라.

### 함수 제한/빌드

* 구현 전후 서버리스 함수 수 확인:

`find api -type f \( -name '*.js' -o -name '*.ts' \) ! -path 'api/_lib/*' | wc -l`

* 12개 초과 시 새 파일 추가를 중단하고 기존 catch-all에 통합하라.
* 다음 파일에 대해 문법 검사를 수행하라:

  * `node --check api/member/[...path].js`
  * `node --check api/practice.js`
  * `node --check script.js`
* `git diff --check`
* 가능한 경우 `npx vercel build`

### 권한/API

* 미로그인 `/api/member/*`는 401을 반환해야 한다.
* owner가 아닌 멤버는 초대 생성이 불가능해야 한다.
* 자기 자신 초대가 불가능해야 한다.
* 이미 멤버인 사용자 초대가 불가능해야 한다.
* 같은 그룹/사용자의 pending 초대 중복 생성이 불가능해야 한다.
* invitee가 아닌 사용자는 초대 수락이 불가능해야 한다.
* accepted/rejected 초대 재수락이 불가능해야 한다.
* `group_members.role`, `group_invites.status` 허용값 외 저장이 불가능해야 한다.

### 데이터 분리

* 사용자 A/B 각각 큐시트 저장 후 자기 큐시트만 로드되어야 한다.
* 사용자 A/B 각각 연습 기록 저장 후 자기 기록만 로드되어야 한다.
* 사용자 A/B 각각 메모 저장 후 자기 메모만 로드되어야 한다.
* 사용자 A의 그룹/초대/메모가 B에게 노출되지 않아야 한다.

### 수동 시나리오

* A 로그인 → 그룹 생성 → B 이메일 초대.
* B 로그인 → 메시지함에서 초대 확인 → 수락.
* B 그룹 목록에 해당 그룹 표시.
* A가 owner인 그룹에서만 초대 UI 동작.
* 로그아웃 직후 이전 사용자의 그룹, 메시지, 메모, 큐시트, 연습 기록이 화면에 남지 않음.
